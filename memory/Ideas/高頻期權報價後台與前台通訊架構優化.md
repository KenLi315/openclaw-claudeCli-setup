# 高頻期權報價：後台 (Backend) 與 前台 Server (Frontend Server) 通訊架構優化

- **建立日期**: 2026-03-27
- **提案人**: Ken
- **狀態**: 技術評估中 (Technical Evaluation)

## 1. 現狀描述 (Current Architecture)
- **數據流**: 報價郵件 -> 後台計算 (十個以上價值) -> 封裝為 JSON -> 發送至前台 Server。
- **後台性能**: 計算速度目前可接受，主要關注後台到前台 Server 的數據交換效率。
- **前台 Server 職責**: 接收 JSON -> 存入資料庫 -> 接收客戶請求 -> 渲染頁面。

## 2. 核心技術挑戰 (Key Concerns)
- **通訊極限**: 每分鐘傳輸大量 JSON 文件的頻率與性能極限。
- **效率瓶頸**: HTTP POST/REST 模式在高頻場景下的連線開銷。

## 3. 秘書 Chris 的優化建議 (Technical Recommendations)

### A. 解決連線極限：從 HTTP 轉向 TCP/gRPC
- **HTTP REST 限制**: 每次傳送 JSON 都要進行 TCP 三向交握，高頻時會造成大量 Port 處於 TIME_WAIT 狀態，導致連線排隊。
- **優化方案**: 使用 **gRPC** 或 **Long-lived TCP Stream**。維持一個長連接，數據像流水一樣直接推過去，消除反覆建立連線的延遲。

### B. 解決傳輸極限：從 JSON 轉向 Protobuf
- **JSON 缺點**: 重複的 Key 名稱（如 "price", "symbol"）佔用大量空間，序列化與反序列化 (Parsing) 耗費 CPU。
- **優化方案**: 使用 **Protocol Buffers (Protobuf)**。
  - 二進制格式，體積僅為 JSON 的 20%-30%。
  - 解析速度快 10 倍以上。這能極大提升前台 Server 接收並寫入資料庫的速度。

### C. 前台 Server 緩存策略：Memory-First
- **瓶頸**: 如果前台 Server 每次收到 JSON 都先寫入硬碟資料庫 (DB)，IO 速度會成為最大瓶頸。
- **優化方案**: 使用 **Redis** 作為快取。後台推過來的價錢先入 Redis，客戶點擊時從 Redis 讀取最快報價，異步再持久化到資料庫。

## 4. 傳輸極限評估
- **千兆網卡環境**: 每秒可傳輸約 100MB 數據。若每個 JSON 為 1KB，理論極限是每秒 10 萬次傳輸。
- **軟體限制**: 真正的極限通常在於「解析 JSON」的 CPU 耗時與「資料庫寫入」的 IOPS。

## 5. 結論
若要支撐更高頻的更新（如每秒多次），應優先棄用 JSON 轉向 **Protobuf + gRPC**。
