# MongoDB 索引優化與架構設計經驗總結 (2026-04-11)

## 核心討論：軟體架構與數據處理效率

### 1. 資料處理模式 (Chatty vs. Chunky I/O)
* **場景**：每日約 20 萬條數據，總量達千萬級。
* **準則**：盡量將篩選邏輯推回資料庫層 (Push down to DB)。
* **折衷**：若 SQL/Query 邏輯無法滿足極度複雜的匹配，應先透過資料庫進行「粗篩」縮小範圍，再批次取回應用程式內存處理，避免頻繁的網絡來回 (Chatty I/O)。

### 2. 複合索引 (Composite Index) 的 ESR 準則
* **定義**：**E (Equality) -> S (Sort) -> R (Range)**。
* **順序關鍵**：等值過濾欄位永遠要放在範圍查詢欄位之前。
* **範圍中斷 (Range Interruption)**：一旦索引中出現範圍查詢 (如 `createDt: { $gte: ... }`)，該欄位之後的所有索引欄位將無法用於快速定位，僅剩過濾功能。
* **實例**：
    * `(underlying, createDt)`：**優**。先定位資產，再連續讀取時間段。
    * `(createDt, underlying)`：**差**。定位時間段後，必須掃描內部所有資產。

### 3. 左綴規則 (Prefix Rule)
* **機制**：查詢條件必須包含複合索引中從左側開始的連續欄位，索引才會生效。
* **案例**：索引為 `{ broker, needQuote, createDt }`。
    * 查詢 `{ broker, needQuote }`：✅ 有效（左綴匹配）。
    * 查詢 `{ underlying, createDt }`：❌ 無效（跳過了 broker）。
* **優化**：利用長索引可以覆蓋短索引的特性，進行索引整合（Consolidation）。

### 4. 文字匹配與模糊搜尋
* **精準匹配**：最快，直接跳轉。
* **前綴匹配 (`/^700/`)**：快，可利用索引進行範圍掃描。
* **包含/後綴匹配 (`/HK/`, `/HK$/`)**：慢，觸發全表/全索引掃描。
* **優化策略**：先用精準匹配欄位（如 `broker`）縮小數據規模（從千萬級降到千級），再對剩下的結果進行模糊搜尋。

### 5. 特殊數據狀態處理
* **空字串 `""` 與缺失欄位**：只要查詢條件中明確包含該欄位，索引即可正常運作。
* **失效風險**：性能失效通常源於程式邏輯在參數缺失時直接「漏掉」了查詢條件中的最左邊欄位。
* **分桶策略 (Bucketing)**：將 `datetime` 降維儲存為 `date_only` 冗餘欄位，將「範圍查詢」轉化為「等值匹配」，可大幅解放後續索引欄位的性能。
